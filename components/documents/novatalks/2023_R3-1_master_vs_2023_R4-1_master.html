1.3 Summary data conclusions

In Create Agents transaction group was observed colossal handling time performance deterioration with new build in the responsible code block.

In Log In transaction group was observed colossal handling time performance improvement with new build in the responsible code block.

In Dashboard requests transaction group was observed significant handling time performance deterioration with new build in the responsible code block.

In Workflow Actions transaction group was observed visible handling time performance deterioration with new build in the responsible code block.

In Workflow Actions transaction group was observed visible deterioration in maximum transaction handling time with new build. Which, in return, points at increased instability of responsible code block and component in overall in handling large amounts of simultaneous requests.

In Agent Messages transaction group was observed visible handling time performance deterioration with new build in the responsible code block.

In Agent Messages transaction group was observed visible deterioration in maximum transaction handling time with new build. Which, in return, points at increased instability of responsible code block and component in overall in handling large amounts of simultaneous requests.

Was observed visible increase of throughput of all transactions with new build. Which, in return, directly points at better performance of Engine and Database pair.

Was observed colossal increase in number of errors with new build.

In Agent Messages transaction group was observed colossal increase in volatility of handling time data dispersion with new build. Which, in return, points at poorer stability of reponsible code block.

In Agent Messages transaction group was observed colossal increase in volatility of maximum transaction handling time data with new build. Which, in return, points at non-efficiency of reponsible code block.

1.7 Test Anomalies

2023_R3-1_master build has stable agent activity results with visible sinusoidal behaviour of transactions elapsed time.

These results are tightly tied with the same kind of Postgres and Engine CPU consumption which has the same visible sinusoidal behaviour.

The most striking 2023_R4-1_master test anomaly was the Log In transaction group and it's two transactions: /app/login and host/

They have a horrifying high spread and elapsed time which isn't seen in any other transaction.

2023_R4-1_master has shown very unstable agent activity results under high load without any visible sinusoidal behaviour which was seen in 2023_R3-1_master build.

The results instability can be advocated to the Engine component's abrupt spike in Memory consumption (3.8 times) without any foreseen reasons, with CPU consumption being on the same level without a spike.

After the Engine component's Memory consumtion spike, the agent's activity started to drop from 100% (all agents are working with conversations) to 10% (90% agents are idle and await for conversation) because of malfunction in autoassignment.

On better note, even with such instability, the Agent Messages transaction group has shown better performance results than 2023_R3-1_master.

Ending Conclusions

Because of instability, the 2023_R4-1_master build is not recomended to use.
