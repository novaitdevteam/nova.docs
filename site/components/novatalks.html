    • Nova.Chats.Core: LDAP authentification NC2-38

    General information 

    â€¢ Does support multiple authentification strategies
    (supports strategies of the same type: LDAP1, LDAP2, etc.)

â€¢ Client can use only authentification that was choosen during registration

â€¢ Authentification provider is shown in Profile Settings
    (Location of providers in ChatWoot database: table "Users", column "Provider", values "LDAP", "Genesys", or "Email")

â€¢ Login and Password change is possible only for "email" authentification provider

    > â€¢ Added "username" field on registration which is shown in profile settings

    > â€¢ If user is registered not with Email provider, all the fields must be filled in (exept of password)

    > â€¢ Agents, administrators and super administrators have access only to their own account information

    > â€¢ One instance to work with multiple clients - BotFlow

    Service configuration modification 

    â€¢ Added configuration in JSON format as an alternative to envirounment variables
    (envirounment variables have the highest priority)

â€¢ Added ability to fill in authentification providers in auth.activeProviders
    (activeProviders - array with provider configurations)

â€¢ Added authentification provider configurations in auth.activeProviders.providers

    Authentification algorithm 

    â€¢ Read from configuration body fields of POST request, where Login and Password are listed

â€¢ Read from configuration activated authentification providers and their configuration 

â€¢ After request on /auth/sign_in/ initiates search of user in ChatWoot database 
    (table "Users", search by "uid" which contains Email)

    - If user is not found, then authentification if failed

    - If user is found, then authentification with provider witten in field user.provider is initiated

    Authentification providers 

     â€¢ "gmail": authentification with Login and Password from ChatWoot database

 â€¢ "genesys": authentification with Genesys ICWS
    (Genesy ICWS agent Login is saved in "user.uid" which can be used instead of Email in UI)

 â€¢ "ldap": authentification through LDAP server
    GET endpoint /auth/active_providers which returns a list of activated providers (array with provider names)

    LDAP parameters 

    â€¢ url: e.g. ldap://localhost:389

â€¢ bindDN: e.g. cn='root'

â€¢ bindCredentials: Password for bindDN

â€¢ searchBase: e.g. o=users,o=example.com

â€¢ searchFilter: LDAP search filter, e.g. (uid={{username)}}. Use literal {{username}} to have the given username used in the search

â€¢ searchAttributes: Optional array of attributes to fetch from LDAP server, e.g. ['displayName', 'mail']. Defaults to undefined, i.e. fetch all attributes

â€¢ usernameField: Field name where the username is found, defaults to username

â€¢ passwordField: Field name where the password is found, defaults to password

â€¢ tlsOptions: Optional object with options accepted by Node.js tls module

    • Nova.Chats: BotAgent development NC2-32

    The task has been extended with NOV-299

    Implementation specifics 

    
 â€¢ BotAgent is a system agent

 â€¢ Realized as an additional Nova.Chats View in the agent workplace

 â€¢ Agent's credentials in Widget are the same as in workplace

 â€¢ Synchronization of actions made in widget and workplace:

    - actions with conversations: pickup, disconnect, transfer 

    - agent's status: online, busy, offline

 â€¢ Telegram plus BotAgent: works without problems on API Inbox

 â€¢ Telegram Inbox plus buttons: doest not have integrated buttons 

 â€¢ Website Inbox plus buttons: widget uses his own endpoints (api/v1/widget) to send messages to ChatWoot

 â€¢ Rich Media are visible in agent's workplace as a title without buttons

 â€¢ Only one bot per inbox

 â€¢ "transfer" word transfers on operators' team

   - default behaviour, if client didn't specify team's or agent's name

   - transfer on agent or team can be initiated at any time

    Rich Media payloads available in ChatWoot 

    
â€¢ "input_select": same as "quick replies" or "inline-buttons" in BotFlow

â€¢ "cards": same as "generic-template" in BotFlow

  (buttons can be added to the "card", but Web Widget would not react to them)

â€¢ "input_email": in BotFlow conform to text message asking to enter Email

  (renders as a field to enter Email in Web Widget)  

â€¢ "article": same as "inline-button" (type: url) or "rich-link" in BotFlow

Inbox webhook will receive events with "content_type" and "content_attributes"
